Open Source Software

(OSS)

  • Firstly, we're notably only talking about software
  • Fundamentally, software where you can freely obtain and use its code, either directly or as part of some other derived work you're building
  • OSI have a formal definition

Free Software

  • This is similar to something you'll hear a lot in similar circles: free software
  • Main proponent is a man called Richard Stallman & the FSF
  • More of a philosophical/political issue than a technical one
  • Primary difference is the use of copyleft licenses, rather than permissive
  • Look at these later, but fundamentally they're attempting to preserve freedom in derivative software too
  • Ensuring that software is not only open-source, but will remain free indefinitely

Free =

  • Gratis

    Libre

  • Free Beer

    Free Speech

  • Use of free might be confusing here
  • English doesn't disambiguate well
  • French and other latin languages explicitly have differentiating words
  • Notably, free software is free as in speech, as in freedom
  • You can do what you like with it, without constraints
  • Not required to be free of cost, but it frequently is anyway

In Practice

  • Most of this is irrelevant day to day
  • Main point: open source means never having to write things yourself
  • Lots of people have written open-source libraries you can use
  • Lots of open-source and free tools that you can use for development and take apart and examine
  • In addition though, you can contribute

Contribute to OSS

  • Contribute to existing projects or release new ones
  • Start with getting involved in existing projects

Why?

  • Do Good
  • Get Good
  • Look Good
  • Few good reasons to do this:
  • Charitable exercise - improve development forever
  • Even true for the smallest change, incremental gains, save a little time for everybody
  • Personal skills
  • Will teach you your tools better, show you common patterns, and is just a good way to practice dev
  • CV points
  • I am impressed by open-source dev on a CV, much easier than most impressive CV things
  • Shows off these new skills, and shows passion and enthusiasm generally
  • Direct business benefit, makes business look good

How?

  • Find a project
  • Find a problem
  • Fix it
  • Hopefully that's somewhat motivating
  • It's still always a bit tricky to see how to get involved though
  • I find that focusing on tools you already use helps
  • Finding something to fix is largely a psychological change
  • Notably not just software
  • Docs, even just typos and so on, traiging bugs
  • Fixing it: the tricky bit, easier on projects that you know well as a user

Github

  1. Fork the repo
  2. Clone locally from your fork
  3. Make changes
  4. Commit
  5. Push to your fork
  6. Open a pull request from your fork to their repo

Github (long version)

  1. Fork the repo
  2. Clone locally from your fork
  3. Add their repo as upstream
  4. Create a branch from master with a relevant name (scrolling-bug)
  5. Make changes
  6. Commit to branch
  7. Fetch & rebase onto upstream master changes
  8. Push to your fork
  9. Open a pull request from your fork to their repo

Elsewhere

  • Mailing list based projects

Things to watch out for

  • Projects on Github not really using Github
  • Previously rejected changes
  • Unmaintained repositories
  • CONTRIBUTING.md
  • Not everything on Github is open-source
